Untangling DDD Concepts

Bounded Contexts

What is a Bounded Context?

A Bounded Context is a central pattern in Domain-Driven Design. More than just a technical container, it is fundamentally a linguistic boundary. It's an explicit line drawn around a specific domain model, and inside that line, all terms and concepts of the model have a single, unified meaning.

"A BOUNDED CONTEXT delimits the applicability of a particular model so that team members have a clear and shared understanding of what has to be consistent... In other CONTEXTS, other models apply, with differences in terminology, in concepts and rules, and in dialects of the UBIQUITOUS LANGUAGE."

- Domain-Driven Design by Eric Evans

Think of it like a cell membrane in biology. The membrane defines what is inside the cell versus outside, and it determines what can pass through. A Bounded Context does the same for your software model:

  • It is a language boundary: It ensures the team's Ubiquitous Language has a precise and unambiguous meaning.
  • It defines the explicit scope of a model.
  • It sets the boundaries for a team's work, usage in specific application parts, and even physical manifestations like codebases.
  • It ensures the model remains pure, consistent, and potent within its applicable area.
  • It prevents confusion when shifting attention to other parts of the system where different models and languages apply.

Why is it Important? The Danger of a "Unified" Model

On any large project, multiple models are always in play. Ignoring this reality leads to buggy, unreliable, and confusing software. When code based on distinct, conflicting models is combined, chaos ensues.

A Cautionary Tale: The 'Charge' Object

Imagine two teams working on a new system:

  • Team A (Bill-Payment): Creates a `Charge` object for paying vendors. Their model includes attributes like "percent deductible" for tax purposes.
  • Team B (Customer-Invoicing): Discovers the existing `Charge` object and decides to reuse it for invoicing customers. They add an "expense code" and use the "posted amount" attribute, which they wanted to call "amount due."

The Result: Unpredictable Failure

Mysterious, nonsensical `Charge` records started appearing in the bill-payment module. The tax report function began to crash because the new `Charge` objects from invoicing didn't have a "percent deductible" value, a field required by the bill-payment module's logic. The two teams had different models for what a "Charge" was, but they didn't realize it.

This story illustrates two key problems that Bounded Contexts solve:

  • Duplicate Concepts: Two different model elements that represent the same concept, leading to synchronization issues and divergent rules.
  • False Cognates (More Dangerous): Using the same term or object (`Charge`) for two different concepts. This leads to teams stepping on each other's code, database contradictions, and confused communication. It's like the Spanish word "embarazada," which looks like "embarrassed" but actually means "pregnant."

By explicitly defining Bounded Contexts, you force these hidden model differences out into the open, allowing you to manage them consciously instead of being surprised by them in production.

Example Scenario: A Booking Application

Let's define the Bounded Context for a new cargo booking application. To do this, we must look at the project as it is, not as we wish it were.

1

The Core Model Team

This team works on the core booking model objects and their persistence. Because they control the database schema and it's driven by their model, the model objects and the database are INSIDE the Bounded Context.

2

The Application Team

This team builds the application that displays and manipulates the model objects. As a primary consumer of the model, their application is also INSIDE the Bounded Context.

3

The Legacy Cargo-Tracking System

Completed bookings must be sent to a legacy system. A decision was made that the new model would depart from the legacy one. Therefore, the legacy system is OUTSIDE the boundary. A translation mechanism must exist at the boundary.

4

The Voyage Scheduling Team

Another team is building a scheduling model. They occasionally share objects with the booking team, but not systematically. This is a RISK. They are working in a separate, undefined Bounded Context. Their informal sharing will cause problems until the relationship between their models is explicitly defined.

By defining the Bounded Context, the teams working inside it gain clarity, and the teams outside gain freedom. Most importantly, it reveals the dangerous, informal sharing with the scheduling team, highlighting an urgent need to define that relationship properly.

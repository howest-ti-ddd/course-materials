Goal

To create a highly extensible and adaptable system by separating minimal, core functionality from extended or custom features, which are implemented as independent plug-in components.

Description

Also known as the Plug-in Architecture, this style is a simple monolithic architecture composed of two main parts: a central core system and multiple plug-in components. It is a natural fit for product-based applications that require customization or have features that can be added or removed, like IDEs (Eclipse), browsers (Chrome), or task managers (Jira).

Properties

  • Partitioning: Uniquely, it can be both technically partitioned (e.g., core has layers) and domain partitioned (e.g., plug-ins for specific business rules).
  • Extensibility: New features can be added as new plug-ins with minimal changes to the core system.
  • Isolation: Application features and volatile code are isolated within plug-ins, improving maintainability and testability.
  • Monolithic Deployment: Typically deployed as a single, self-contained unit.

Core Concept: Core & Plug-ins

The architecture is defined by the relationship between the core system and its plug-ins.

1. The Core System

Contains only the minimal functionality required to run the system (the "happy path"). It knows how to find and invoke plug-ins but is not concerned with their specific implementation details.

2. Plug-in Components

Independent, standalone components that contain specialized processing, additional features, or custom logic. Ideally, plug-ins do not depend on each other.

In-Depth Details

Registry & Contracts

The core system uses a Registry (like a HashMap or a discovery tool) to know what plug-ins are available. Communication happens via a standardized Contract (like a Java Interface) that defines what data is sent to the plug-in and what is expected back.

Plug-in Communication

Plug-ins can be connected to the core system in different ways, each with trade-offs between simplicity and scalability.

Point-to-Point (Simple)

Plug-ins are shared libraries (JARs, DLLs) or packages within the same codebase. Simple and fast.

devicePlugin.assess();

Remote Access (Complex)

Plug-ins are deployed as separate services, communicating via REST or messaging. More scalable but complex.

POST /assess/iphone6s

Data Topologies

Typically, the core system manages a central database. Plug-ins should not access this directly. Instead, the core passes necessary data to them. Plug-ins can, however, own their own separate, specialized data stores.

Deployment & Cloud

Deployment options are generally coarse-grained because the architecture is monolithic. You can deploy the entire application to the cloud, but separating the core and plug-ins across on-premise and cloud environments is challenging and can introduce significant network latency, impacting performance.

Team Topology

This architecture maps well to team structures, allowing for a clear separation of concerns.

A stream-aligned team can own the core system, while specialized enabling or complicated-subsystem teams can focus on developing and maintaining individual plug-ins for experiments or complex features.

Usage Guide

When to Use It

  • Product-based applications (IDEs, browsers, Jira, Jenkins).
  • Systems with frequently changing rules that vary by context (e.g., insurance rules by state).
  • Applications where third parties can add features.
  • When you need to isolate volatile code from a stable core.

When Not to Use It

  • When the core system itself is volatile and changes constantly.
  • When plug-ins have many dependencies on each other.
  • For systems requiring high scalability and fault tolerance.

Usecase Example: Tax Software

In tax preparation software, the main tax form (e.g., Form 1040) acts as the core system. It defines the overall process. Each supporting schedule or worksheet (Schedule A for deductions, Schedule C for business income, etc.) is implemented as a separate plug-in. When tax laws change, only the affected plug-in needs to be updated, not the entire system.

Goal

To structure an application as a series of discrete, single-purpose, and independent components (filters) that process data sequentially through communication channels (pipes). This promotes reuse and simplifies complex processing tasks.

Description

Also known as the Pipes and Filters architecture, this is a fundamental style where functionality is broken down into separate processing steps. It's famously the underlying principle of Unix shell commands (`grep | sort | uniq`), and is common in functional programming and ETL (Extract, Transform, Load) tools.

Properties

  • Technically Partitioned: Components (filters) are separated by their specific function or task.
  • Unidirectional Flow: Data moves in one direction through the pipes, from one filter to the next.
  • Stateless Filters: Filters are typically self-contained, stateless, and independent of other filters.
  • Monolithic or Distributed: While often monolithic, filters can be deployed as distributed services or serverless functions.

Core Concept: Pipes and Filters

The architecture has two component types: **Filters**, which perform the work, and **Pipes**, which connect them. There are four types of filters:

1. Producer (Source)

The starting point of the pipeline. It creates data and sends it to the first pipe.

2. Transformer (Map)

Accepts input, transforms it (e.g., converts format, enhances data), and passes it on.

3. Tester (Reduce)

Tests data against criteria and decides whether to pass the data on or terminate a branch.

4. Consumer (Sink)

The termination point. It receives the final data, and might save it to a database or display it on a UI.

Common Risks

  • Overloaded Filters: A filter doing too many things violates the single-purpose principle.
  • Bidirectional Communication: Pipes should be one-way. If back-and-forth communication is needed, this is likely the wrong architecture.
  • Error Handling: It can be difficult to manage fatal errors and determine how to recover once a pipeline is running.
  • Contract Management: Changing the data format in a pipe requires careful coordination to avoid breaking downstream filters.

Governance

Governance focuses on ensuring filters adhere to their specific roles. A common technique is to use tags (annotations in Java, attributes in C#) to programmatically label each filter's type (`@Filter(FilterType.PRODUCER)`), making its purpose clear to developers.

Deployment & Cloud

This style is highly suited for the cloud. While it can be a simple monolith, its modularity shines when filters are deployed as individual serverless functions (like AWS Lambda) and orchestrated with a workflow service (like AWS Step Functions).

Team Topology

Like the Layered style, the Pipeline architecture is generally independent of team topologies and works well with any team configuration due to its high modularity and clear, single-purpose components.

Whether it's a stream-aligned team owning an entire pipeline, an enabling team adding a new analysis filter, or a complicated-subsystem team focusing on one complex filter, the architecture's structure supports various team models effectively.

Usage Guide

When to Use It

  • Simple, one-way processing tasks.
  • ETL (Extract, Transform, Load) jobs.
  • Data processing and transformation pipelines.
  • Projects with tight budget and time constraints.

When Not to Use It

  • Systems that require high scalability or fault tolerance (in a monolithic deployment).
  • Scenarios with complex, back-and-forth interactions between components.
  • Non-deterministic workflows with many branching conditions.

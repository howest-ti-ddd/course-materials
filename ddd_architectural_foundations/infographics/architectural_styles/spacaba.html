Goal

To overcome scalability and concurrency limitations by removing the central database as a transactional bottleneck, enabling near-infinite scalability for applications with high, variable, and unpredictable user loads.

Description

Space-Based Architecture (also known as "cloud-native architecture") is a high-performance style that achieves extreme scalability by using replicated in-memory data grids. Application data is kept in-memory and replicated across multiple parallel processing units. Updates are sent asynchronously to a database, which acts as a system of record rather than a real-time transactional constraint.

Properties

  • In-Memory Data Grids: Application data is held and processed in memory for maximum speed.
  • Processing Units: Self-sufficient units containing application logic and a portion of the data grid.
  • Virtualized Middleware: Manages requests, data synchronization, and processing unit lifecycle.
  • Asynchronous Persistence: Data is written to the database asynchronously via data pumps, ensuring the database does not slow down user transactions.
  • High Elasticity: Processing units can be started and stopped dynamically based on user load.
  • Technical Partitioning: A given domain is represented by different technical components (processing units, data pumps, etc.).

Core Concept: A Distributed In-Memory System

User requests are handled by a set of identical Processing Units. Each unit has an in-memory replicated cache. When data is updated in one unit, it's synchronized to all others. The database is updated in the background.

In-Depth Details

Data Caching Models

The choice of caching is critical. It's a trade-off between performance, fault tolerance, and data consistency.

Replicated Cache

Each unit holds a full copy. Offers max performance and high fault tolerance. Best for smaller, less frequently updated data.

Distributed Cache

A central cache server holds the data. Offers max data consistency but lower performance and a single point of failure. Best for large or frequently updated data.

Data Synchronization

Data is persisted to the database via a "fire-and-forget" mechanism, ensuring application responsiveness.

Deployment & Cloud

This style is a natural fit for cloud environments, which excel at providing dynamic, elastic infrastructure. A powerful hybrid model involves running the elastic processing units in the cloud while keeping the master database on-premises for security and control, with synchronization happening via data pumps.

Team Topology

Due to the many complex, cross-cutting technical components (data grids, message queues, data writers), this architecture often works well with specialized teams.

Complicated-Subsystem and Enabling Teams are a great fit. A dedicated team can own and optimize the complex caching and data synchronization infrastructure, allowing other teams to focus on business logic within the processing units.

Usage Guide

When to Use It

  • Applications with extremely high and spiky user loads (e.g., >10,000 concurrent users).
  • When scalability and responsiveness are the most critical requirements.
  • For systems where eventual consistency of the database is acceptable.
  • Online bidding/auction sites, concert ticket sales, and social media event platforms.

When Not to Use It

  • Simple applications with low to moderate user traffic.
  • When strict, immediate transactional consistency with the database is required.
  • If the development team lacks expertise in distributed systems and caching.
  • If the budget is tight; this is an expensive and complex architecture to build and maintain.

Usecase Example: Concert Ticketing System

A concert ticketing system experiences extremely low traffic for weeks, then suddenly spikes to tens of thousands of concurrent users the moment tickets for a popular artist go on sale. A traditional architecture would collapse under this load as every user tries to query and update seat availability in the database simultaneously.

How Space-Based Architecture Helps

Elasticity on Demand

The deployment manager instantly scales up from a few processing units to hundreds to handle the ticket-buying frenzy.

In-Memory Speed

Seat availability is managed in the replicated in-memory grid, providing millisecond response times without hitting the slow database.

Asynchronous Sales

Completed ticket sales are pushed to a message queue, to be written to the database later. The user gets their confirmation instantly.

Scale Down After

Once the tickets sell out and the load drops, the extra processing units are automatically shut down to save costs.
